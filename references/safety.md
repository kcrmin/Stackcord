# Safety boundaries

- Inspect files, Git, worktrees, submodules, and configured tools before asking discoverable facts.
- Never hide fetch, pull, rebase, stash, reset, clean, force-push, pointer movement, install, external write, merge, publication, or production mutation.
- Plan local mutations before applying them. Require exact target confirmation for destructive, credential, production, or irreversible actions.
- Use conventional branch and commit names without AI markers. Claims express time-bounded intent; they do not lock files.
- Treat conflict outcomes as `clear`, `coordinate`, `block`, or `unknown`. Restore visibility before declaring unknown work safe.
- Keep Git DBML canonical. Quarantine dbdiagram and external UI input until semantic, license, security, accessibility, and authority review completes.
- Do not expose operation IDs, claims, receipts, or `.harness/` internals in normal explanations. Summarize their meaning when they matter.
- Detect external tools first, explain trade-offs, and connect only the selected real tool. One source owns live task status.
